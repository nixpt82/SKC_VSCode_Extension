## app.json Locales Configuration

### What is `locales` in app.json?

The `locales` property defines which languages your extension officially supports:

```json
{
  "name": "My Extension",
  "features": [
    "GenerateCaptions",
    "TranslationFile"
  ],
  "locales": ["en-US", "fr-FR", "de-DE"]  ← Supported languages
}
```

### How It Works:

1. **Base Locale** (usually en-US):
   - Your AL code's Labels are in this language
   - Compiler generates `MyApp.g.xlf` from this

2. **Target Locales** (e.g., fr-FR, de-DE):
   - You create `MyApp.fr-FR.xlf` and `MyApp.de-DE.xlf`
   - These files translate the base locale strings

3. **When Installing**:
   - BC checks user's language setting
   - If user is French → Uses MyApp.fr-FR.xlf
   - If user is German → Uses MyApp.de-DE.xlf
   - If XLF missing → Falls back to en-US

### Default Behavior:

If you don't specify `locales` in app.json:
- Defaults to `["en-US"]`
- Only English is officially supported
- But you can still add XLF files for other languages

### Best Practice:

**Option 1: List Only Supported Languages**
```json
"locales": ["en-US", "fr-FR", "de-DE"]
```
- Clear communication of what you support
- Users know what to expect

**Option 2: Don't Specify (Flexible)**
```json
// No locales property
```
- Allows any language XLF to be added
- More flexible for community translations
- Still works - BC will find XLF files automatically

## Translation File Naming

### Naming Pattern:
```
{AppName}.g.xlf           // Source/Generated (en-US usually)
{AppName}.{locale}.xlf    // Target language translations
```

### Example for "Companial Interface":
```
Translations/
├── Companial Interface.g.xlf        // Generated by build
├── Companial Interface.fr-FR.xlf    // French
├── Companial Interface.de-DE.xlf    // German
├── Companial Interface.pt-PT.xlf    // Portuguese
└── Companial Interface.es-ES.xlf    // Spanish
```

### Locale Codes Must Match:
- File name: `MyApp.fr-FR.xlf`
- Inside XLF: `<file target-language="fr-FR">`
- BC language setting: `fr-FR`
